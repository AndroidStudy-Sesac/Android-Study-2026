> ### 핵심 요약
> - SOLID 원칙: 코드를 썩지 않게, 그리고 유연하게 유지하기 위한 선배들의 5가지 피땀 어린 철학입니다. <br/> <b>(특히 S와 D가 안드로이드 실무의 핵심입니다.)</b>
> - 디자인 패턴: 코딩하다 마주치는 뻔한 문제들에 대한 '모범 정답지'입니다.
> - DI (의존성 주입): 현대 안드로이드 개발의 숨쉬기 같은 존재입니다. 부품을 내부에서 직접 만들지 않고 밖에서 꽂아주어, 결합도를 낮추고 테스트를 가능하게 합니다.
> - Repository & Observer: 데이터의 출처를 숨기고, 데이터가 변할 때만 UI에 알림을 쏴주는 환상의 콤비입니다.

## 1. SOLID 원칙 (절대 무너지지 않는 코드 짓기)
  객체지향 프로그래밍의 5대 원칙입니다.

  | **원칙** | **의미** | **안드로이드 실무 적용 예시** |
  | --- | --- | --- |
  | **S** (Single Responsibility) <br/>**단일 책임 원칙** | 클래스는 <b>단 한 개의 책임(역할)</b>만 가져야 한다. | `Activity`는 화면만 그리고, 데이터 통신은 `Repository`가, 상태 관리는 `ViewModel`이 합니다. (만능 God Class 방지) |
  | **O** (Open-Closed) <br/> 개방-폐쇄 원칙 | 확장에는 열려 있고, 수정에는 닫혀 있어야 한다. | 새로운 기능을 추가할 때 기존 코드를 뜯어고치는 게 아니라, 새로운 클래스를 만들어 상속/구현하게 합니다. |
  | **L** (Liskov Substitution) <br/> 리스코프 치환 원칙 | 부모 객체를 자식 객체로 갈아 끼워도 프로그램은 정상 작동해야 한다. | `RecyclerView.Adapter`를 상속받아 커스텀 어댑터를 만들어도, 기존 리사이클러뷰 동작이 망가지면 안 됩니다. |
  | **I** (Interface Segregation) <br/> 인터페이스 분리 원칙 | 자신이 쓰지 않는 메서드에 의존하지 않도록 인터페이스를 잘게 쪼개라. | 덩치 큰 인터페이스 하나보다, 클릭 리스너, 스와이프 리스너 등 목적별로 인터페이스를 분리합니다. |
  | **D** (Dependency Inversion) <br/> **의존성 역전 원칙** | 구체적인 구현(로컬 DB 객체)이 아닌, **추상적인 인터페이스에 의존**해야 한다. | ViewModel은 `Retrofit`이나 `Room` 같은 구체적 기술을 몰라야 합니다. 오직 `Repository Interface`만 바라봅니다. |
  
💡 실무 팁
- 모두 완벽할 필요는 없습니다
  - 실무에서 이 5가지를 100% 지키려다 보면 파일 개수가 수백 개로 늘어나 오버엔지니어링이 되기 쉽습니다.
  - 하지만 <b>S(단일 책임)</b>와 <b>D(의존성 역전)</b>만큼은 무조건, 숨 쉬듯이 지켜야 유지보수 지옥을 피할 수 있습니다.

## 2. 안드로이드 필수 디자인 패턴
### [1] Singleton (싱글톤 패턴)
- 목적: 앱이 켜져 있는 동안 메모리에 단 1개의 객체만 올려놓고 돌려쓰고 싶을 때 사용합니다.
- 실무 예시: 네트워크 통신을 담당하는 Retrofit 클라이언트, 로컬 DB인 Room Database 인스턴스.
> 🚨 실무 주의사항
>
> 너무 편하다고 아무 데이터나 싱글톤으로 만들면 메모리가 낭비되고 꼬입니다. <br/>
> 특히 내부에 Activity Context를 들고 있으면 치명적인 메모리 릭이 발생하니 절대 주의하세요!

### [2] Dependency Injection (DI, 의존성 주입)
- 목적: 객체가 사용할 부품을 스스로 만들지 않고, 외부에서 조립해서 꽂아주는(Inject) 방식입니다.
- 실무 예시: Hilt나 Dagger 라이브러리를 써서 ViewModel에 Repository를 자동으로 꽂아줍니다.
> 💡 왜 쓸까? 부품을 갈아 끼우기 쉬워져서 테스트 코드를 짜기가 엄청나게 편해집니다. (서버 대신 가짜 Mock 서버를 꽂아넣기 등)

### [3] Repository (리포지토리 패턴)
- 목적: 데이터가 어디서 오는지(서버인지, 로컬 캐시인지, DB인지) 출처를 숨기고, 데이터를 깔끔하게 건네주는 창구 역할입니다.
- 실무 장점: 네트워크가 끊겼을 때 로컬 DB의 데이터를 보여주는 복잡한 분기 처리를 ViewModel이나 UI는 전혀 모르게 뒤에서 몰래 처리할 수 있습니다.

### [4] Observer (옵저버 패턴)
- 목적: 데이터가 변경될 때마다, 그걸 쳐다보고(구독하고) 있던 녀석들에게 "데이터 바뀌었어!"라고 자동으로 방송을 쏴주는 패턴입니다.
- 실무 예시: LiveData, StateFlow. UI는 이 방송을 듣고 있다가 알림이 오면 화면을 새로고침만 하면 됩니다.

### [5] Builder & Factory (빌더 & 팩토리 패턴)
- Builder: 햄버거에 토마토 빼고 치즈 추가하듯, 복잡한 객체를 옵션 붙여가며 차곡차곡 조립할 때 씁니다. (예: AlertDialog.Builder, Retrofit.Builder)
- Factory: 객체를 생성하는 복잡한 로직을 공장(Factory) 클래스에 몰아넣어 생성 책임을 분리합니다. (예: 생성자에 매개변수가 필요한 ViewModel을 만들 때 쓰는 ViewModelProvider.Factory)
