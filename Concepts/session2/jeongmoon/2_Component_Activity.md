> ### 핵심 요약
> - Activity의 본질: 사용자와 상호작용하는 단일 화면(UI)이자, 앱의 가장 기본적인 진입점입니다.
> - Task와 Back Stack: 안드로이드 시스템은 열려있는 Activity들을 '태스크(Task)'라는 작업 묶음 안에서 '백 스택(Back Stack)'이라는 바구니에 차곡차곡 쌓아(**LIFO**) 관리합니다.
> - Launch Mode (시작 모드): 똑같은 화면을 중복해서 띄울 것인지, 아니면 기존 화면을 재활용할 것인지 결정하는 아주 중요한 규칙입니다.

## 1. Task와 Back Stack (화면이 쌓이는 원리)
안드로이드는 사용자가 여러 앱을 오가며 작업할 수 있도록 화면들을 스택(Stack) 구조로 관리합니다.
- 동작 방식 (**Last-In, First-Out**)
  - A 화면에서 B 화면을 켜면, B가 A 위에 쌓입니다.
  - 여기서 '뒤로 가기' 버튼을 누르면 가장 위에 있는 B가 파괴(Pop)되고 A가 다시 나타납니다.
- 문제점
  - 아무런 설정 없이 버튼을 마구 누르면, 똑같은 '결제 화면'이 스택에 10개씩 쌓이는 대참사가 발생합니다.
  - 사용자가 뒤로 가기를 10번 눌러야 빠져나올 수 있습니다.
  - 이를 방지하기 위해 **Launch Mode**를 사용합니다.

---

## 2. Launch Mode (시작 모드) 완벽 정리
`AndroidManifest.xml`의 <activity> 태그 안에 `android:launchMode` 속성으로 지정합니다.

  | **모드 (Mode)** | **동작 방식** | **실무 사용처 및 특징** |
  | --- | --- | --- |
  | **`standard`** | **(기본값)** 호출할 때마다 스택 위에 **무조건 새로운 Activity 인스턴스를 생성**하여 쌓습니다. | 일반적인 대부분의 화면 (예: 상품 상세 페이지 - 여러 상품을 연달아 볼 때 각각 쌓여야 함) |
  | **`singleTop`** | 호출된 Activity가 **스택의 가장 위(Top)에 이미 존재한다면, 새로 만들지 않고 기존 객체를 재활용**합니다. (이때 `onNewIntent()`가 호출됨) | **푸시 알림 클릭 시 이동하는 화면.** (채팅방에 있는데 알림을 눌렀다고 똑같은 채팅방 화면이 하나 더 열리면 안 되기 때문) |
  | **`singleTask`** | **태스크 내에 오직 1개의 인스턴스만 존재**합니다. 만약 스택 중간에 이미 존재한다면, **그 위에 쌓여있던 다른 화면들을 전부 파괴(Clear)**하고 해당 화면을 최상단으로 끌어올립니다. | **앱의 메인 화면(Home).** (어느 화면에 있든 '홈으로 가기'를 누르면, 그동안 쌓인 자잘한 화면들을 다 날려버리고 메인 화면 하나만 깔끔하게 남겨야 함) |
  | **`singleInstance`** | `singleTask`와 유사하지만, **아예 혼자만 있는 독방(새로운 Task)을 만들어서** 그 안에 격리시킵니다. 다른 Activity와 섞이지 않습니다. | **전화 수신 화면, 알람 울림 화면.** (앱의 일반적인 흐름과 완전히 분리되어야 하는 특수한 시스템 레벨의 화면) |

💡 실무 팁
- 코드(Intent Flag)로 제어하기
  - 매니페스트에 고정해 두는 방법도 있지만, 실무에서는 코드로 화면을 이동할 때 Intent에 깃발(Flag)을 달아 동적으로 제어하는 방식을 더 많이 씁니다
    - **`Intent.FLAG_ACTIVITY_CLEAR_TOP`** : singleTask와 유사한 효과
    - **`Intent.FLAG_ACTIVITY_SINGLE_TOP`** : singleTop과 동일한 효과
- singleInstance 남용 금지
  - 이 모드를 쓰면 앱의 '뒤로 가기' 흐름이 완전히 끊어져 버립니다.
  - 예기치 못한 UX(사용자 경험)가 발생하므로, 특수 목적이 아니면 사용을 지양해야 합니다.
    (예. 사용자는 뒤로 가기를 눌렀는데 바탕화면으로 튕겨 나가는 등)
- 데이터 갱신 주의 (onNewIntent)
  - singleTop이나 singleTask로 인해 화면이 새로 생성되지 않고 재활용될 때는 `onCreate()`가 다시 호출되지 않습니다.
  - 대신 `onNewIntent()` 함수가 호출되므로, 이전 데이터가 계속 남아있지 않도록 이 함수 안에서 반드시 새로운 데이터를 받아 화면을 갱신해 주어야 합니다.
