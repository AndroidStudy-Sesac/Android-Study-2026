> ### 핵심 요약
> - Service의 본질: 화면(UI) 없이 백그라운드에서 오래 걸리는 작업을 수행하는 '얼굴 없는 일꾼'입니다.
> - 메인 스레드 동작: Service는 기본적으로 앱의 메인 스레드에서 동작하므로, 무거운 작업 시 반드시 내부에서 코루틴이나 별도 스레드를 생성해야 합니다. (ANR 발생 주의)
> - 현대 안드로이드의 제약: 배터리 절약을 위해 안드로이드 시스템은 순수 백그라운드 서비스를 가차 없이 종료시킵니다. 현재는 상단바 알림이 필수인 Foreground Service나 WorkManager로 대체되었습니다.

## 1. Service의 3가지 주요 유형
안드로이드의 서비스는 동작 방식과 사용자와의 상호작용 여부에 따라 크게 세 가지로 나뉩니다.

  | **유형** | **동작 방식 및 특징** | **실무 사용처** |
  | --- | --- | --- |
  | **Background Service** | 사용자에게 보이지 않는 상태로 뒤에서 작업을 수행합니다. (안드로이드 8.0 이후 실행이 엄격히 제한됨) | 실무에서 거의 사용하지 않습니다. (최신 앱에서는 `WorkManager`로 대체됨) |
  | **Foreground Service** | 사용자가 앱을 사용하지 않아도 시스템 리소스를 최우선으로 보장받으며 동작합니다. **반드시 상태바(Notification)에 작업 중임을 알리는 알림을 띄워야 합니다.** | 음악/오디오 재생 앱, 만보기 및 GPS 러닝 트래커, 대용량 파일 다운로드 |
  | **Bound Service** | Activity 등 다른 컴포넌트가 서비스와 연결(Bind)되어 데이터를 주고받는 '클라이언트-서버' 구조입니다. | 앱 내부에서 음악 재생을 세밀하게 제어하거나, 지속적인 센서 데이터를 UI로 스트리밍할 때 |

## 2. Service 생명주기 (Lifecycle)
Service는 호출하는 방식에 따라 두 가지 생명주기 흐름을 가집니다.
- startService() (Started Service)
  - `onCreate()` -> `onStartCommand()` 흐름을 탑니다.
  - 호출한 컴포넌트(Activity)가 파괴되어도, 서비스는 스스로 종료(stopSelf())하거나 시스템이 메모리가 부족해 죽일 때까지 계속 돌아갑니다.
- bindService() (Bound Service):
  - `onCreate()` -> `onBind()` 흐름을 탑니다.
  - 서비스와 연결된 컴포넌트들이 모두 연결을 해제(unbindService())하면, 서비스도 소임을 다하고 소멸(`onUnbind` -> `onDestroy`)됩니다.

🚨 실무 팁
- Service는 별도의 스레드가 아닙니다
  > "서비스는 백그라운드에서 도니까 네트워크 통신을 그냥 넣어도 되겠지?"라고 생각하면 100% ANR(Application Not Responding, 앱 멈춤) 크래시가 발생합니다.
  - Service도 앱의 '메인 스레드(UI 스레드)' 위에서 돌아갑니다.
  - 긴 작업(DB 통신, 네트워크 다운로드)을 하려면 서비스 내부에서 별도의 스레드(Coroutine 등)를 열어서 처리해야 합니다.

- Foreground Service 타입 명시 (최신 안드로이드 정책)
  - 안드로이드 14(API 34)부터는 매니페스트에 단순히 권한만 주는 것을 넘어, <br/>
  서비스가 '어떤 목적'으로 돌아가는지(예: location, mediaPlayback, dataSync 등) Foreground Service Type을 반드시 명시해야 합니다.
  - 누락 시 런타임 에러(SecurityException)가 발생하며 앱이 강제 종료됩니다.

## 3. 백그라운드 작업 가이드
현대 안드로이드 개발에서는 무작정 Service를 띄우지 않고, 목적에 맞는 도구를 선택하는 것이 아키텍처 설계의 핵심입니다.

| **작업의 성격** | **권장되는 도구** | **실무 예시** |
| --- | --- | --- |
| **즉시 실행 & 짧은 작업** <br/>(사용자가 화면을 보고 있을 때) | **Kotlin Coroutines** | `ViewModel` 안에서 코루틴을 열어 API 데이터를 받아오는 일반적인 상황. |
| **즉시 실행 & 긴 작업** <br/>(사용자가 앱을 꺼도 계속 유지) | **Foreground Service** | 멜론 음악 재생, 넷플릭스 영상 다운로드처럼 사용자가 '진행 중'임을 상단바를 통해 명확히 인지해야 하는 작업. |
| **지연 가능 & 보장된 작업** <br/>(조건이 맞을 때 반드시 실행) | **WorkManager** | "와이파이가 연결되고 기기가 충전 중일 때 100MB 로그를 서버에 전송해" 같은 조건부 백그라운드 작업. |

💡 실무 팁
- 과거에는 알람 앱이나 푸시 수신 후 처리를 위해 Background Service를 무분별하게 사용했습니다.
- 하지만 현재는 배터리 최적화(Doze 모드) 정책으로 인해, 순수 백그라운드 서비스는 시스템이 몇 분 내로 죽여버립니다.
- 따라서 '사용자가 즉시 눈치채야 하는 작업(음악, 내비게이션)'이 아니라면, 대부분의 백그라운드 처리는 **WorkManager**로 위임하는 것이 모던 안드로이드의 정답입니다.
