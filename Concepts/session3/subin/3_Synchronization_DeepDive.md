# 동기화 심화

## Race condition · Deadlock · Starvation · Atomic · synchronized · Mutex · Semaphore

## 전체 핵심 요약

- 동기화는 “비동기” 자체가 아니라, **동시에 접근할 때 데이터가 깨지는 문제**를 해결하는 기술
- 문제가 생기는 조건은 거의 항상 “공유 상태(shared mutable state)” + “동시 실행”
- 해결 전략은 2가지로 정리됨:
    1. **공유 상태를 없애기(불변/단일 스레드/메시지 패싱)**
    2. 공유가 불가피하면 **동기화 도구(락/세마포어/원자 연산)** 사용

---

# 1) 동시성 버그 3종 세트

## 1-1) Race condition (경쟁 조건)

- 여러 작업이 동시에 같은 데이터를 읽고/쓰면서 결과가 **실행 타이밍에 따라 달라지는** 문제
- 예: `count++`를 동시에 실행하면 증가가 누락될 수 있음

**핵심 원인**

- 읽기 → 수정 → 쓰기가 원자적(atomic)으로 보장되지 않음

---

## 1-2) Deadlock (교착 상태)

- 서로가 서로의 락을 기다리면서 **영원히 진행이 멈추는** 상태

**대표 상황**

- A는 Lock1 잡고 Lock2 기다림
- B는 Lock2 잡고 Lock1 기다림

---

## 1-3) Starvation (기아)

- 특정 작업이 계속 우선순위/락 경쟁에서 밀려 **실행 기회를 거의 못 얻는** 문제
- Deadlock처럼 “완전 멈춤”은 아니지만, 사실상 멈춘 것처럼 보일 수 있음

---

# 2) 해결 도구(개념): Atomic / synchronized / Mutex / Semaphore

## 2-1) Atomic(원자성)

### 핵심

- 한 번에 하나의 연산처럼 중간 상태가 보이지 않게 처리되는 성질
- 원자성이 필요한 대표 케이스: 카운터 증가, 플래그 전환, 상태 업데이트

### 실무 감각

- 단순 카운터/플래그면 “락”보다 원자 연산이 더 가벼울 수 있음
- 하지만 복합 로직(여러 변수 일괄 업데이트)은 원자 연산만으로 부족해서 락이 필요

---

## 2-2) synchronized (자바/스레드 락)

### 핵심

- 임계 구역(critical section)을 “한 번에 하나”만 들어가게 하는 모니터 락
- 스레드 기반 코드에서 자주 사용

### 실무 감각(안드로이드)

- Java/레거시 코드, 단순 동기화에 등장
- UI 스레드에서 synchronized로 오래 잡으면 프리징 위험

---

## 2-3) Mutex (코루틴 친화적 락)

### 핵심

- 코루틴에서 “한 번에 하나만” 임계 구역 접근을 보장
- `withLock`은 suspend-friendly (스레드를 블로킹하지 않게 설계)

### 실무 감각

- 코루틴 기반 앱에서 synchronized 대신 Mutex를 선택하는 경우가 많음
- 특히 “네트워크 토큰 갱신” 같은 동시 호출 방지에 자주 씀

---

## 2-4) Semaphore (동시 접근 N개 제한)

### 핵심

- “동시에 N개까지만” 허용
- Mutex(1개 허용)를 일반화한 형태

### 실무 감각

- 네트워크 동시 요청 제한(서버 보호/레이트 리밋)
- 이미지/파일 처리 동시 실행 개수 제한(메모리/CPU 보호)

---

# 3) 암기 포인트

## 패턴 A) “단일 실행 보장” — 중복 호출 막기 (Mutex)

- refresh token이 동시에 여러 번 호출되면 안 되는 케이스
- 결제/주문 같은 “한 번만 처리되어야 하는 로직”에도 개념적으로 동일

**핵심 감각**

- “동시에 들어오면 안 된다” → Mutex

---

## 패턴 B) “동시성 제한” — 폭주 방지 (Semaphore)

- 이미지 업로드 100개를 한 번에 다 보내면 터짐 → 3개만 동시 처리
- 검색 API 동시 요청 폭주 제한

**핵심 감각**

- “동시에 너무 많이 돌리면 안 된다” → Semaphore

---

## 패턴 C) “공유 상태 제거” — 가장 강력한 해결

- immutable data + 새 객체로 교체(copy)
- 단일 스레드로 상태를 몰아넣기(메인/전용 스레드)
- 메시지 기반(채널/이벤트 큐)으로 직렬화

**핵심 감각**

- 가능하면 락보다 “공유 자체를 줄이는 설계”가 유지보수에 유리

---

# 4) 주의점

## 4-1) 락 범위를 최소화

- 임계 구역이 길면 성능 저하 + 기아/대기 증가
- “데이터 읽기/갱신 핵심 부분”만 잠그고, I/O는 락 밖에서

## 4-2) Deadlock 방지 규칙

- 락을 여러 개 잡아야 한다면 **항상 동일한 순서로** 잡기(규칙화)
- 가능하면 한 번에 하나만 잡도록 구조 변경

## 4-3) Cancellation과 Mutex

- 코루틴 취소가 섞이면 락 해제/정리 보장이 중요
- `try/finally` 또는 `withLock` 사용이 안전한 이유

---

# 5) 정리

- Race condition: 타이밍에 따라 결과가 바뀜 → 원자성/락 필요
- Deadlock: 서로 락 기다리며 멈춤 → 락 순서 규칙/구조 변경
- Starvation: 계속 밀려 실행 못 함 → 락 범위 최소화/정책 조정
- Mutex: 1개만 허용(코루틴) / Semaphore: N개 허용 / synchronized: 스레드 락(레거시)
